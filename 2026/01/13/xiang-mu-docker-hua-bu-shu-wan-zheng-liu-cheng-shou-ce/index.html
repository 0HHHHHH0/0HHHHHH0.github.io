📚 WangLab 项目 Docker 化部署完整流程手册

📋 项目信息

  • 项目名: WangLab 科研平台
  • 技术栈: React + Node.js + MongoDB
  • 部署目标: 飞牛 NAS (Docker 环境)
  • 特殊需求: MongoDB 4.4、端口 15173、Google Gemini API

📁 目录结构

WangLab/                          # 项目根目录
├── apps/                         # React 应用模块
├── components/                   # 共享组件
├── server/                       # Node.js 后端
│   ├── index.js                  # 后端入口
│   ├── package.json              # 后端依赖
│   ├── package-lock.json         # 后端依赖锁文件
│   └── models/                   # MongoDB 模型
├── services/                     # 前端业务服务
├── Dockerfile                    # Docker 构建文件
├── docker-compose.yml           # Docker 编排配置
├── package.json                  # 前端依赖
├── vite.config.ts               # 前端构建配置
└── nginx.conf                   # Nginx 配置

🚀 完整部署流程

第一阶段：本地准备和测试(笔记本ubuntu)

1. 检查项目结构

# 确认关键文件存在
cd ~/Downloads/WangLab
ls -la Dockerfile docker-compose.yml server/package.json

Dockerfile文件如下：


# Stage 1: 构建前端 (Build Frontend)
FROM node:20-alpine AS builder

# 设置工作目录
WORKDIR /app

# 复制前端依赖配置
COPY package.json package-lock.json* ./

# 安装前端依赖
RUN npm install

# 复制所有源代码
COPY . .

# 执行构建 (生成 /app/dist)
RUN npm run build

# Stage 2: 生产环境 (Production Server)
FROM node:20-alpine

# 设置工作目录为后端目录
WORKDIR /app/server

# 复制后端依赖配置
COPY server/package.json server/package-lock.json* ./

# 安装后端依赖 (仅生产环境依赖)
RUN npm install --production

# 复制后端源代码
COPY server/ .

# 从 Stage 1 复制构建好的前端静态文件到 /app/dist
# 注意：server/index.js 中配置的是 path.join(__dirname, '../dist')
COPY --from=builder /app/dist ../dist

# 暴露端口 (后端默认 5000)
EXPOSE 5000

# 启动命令
CMD ["node", "index.js"]

docker-compose.yml文件如下：



version: '3.8'

services:
  # 1. 数据库服务 (MongoDB)
  mongodb:
    image: mongo:4.4
    container_name: wanglab-db
    restart: always
    volumes:
      - ./mongodb_data:/data/db # 数据持久化到当前目录
    ports:
      - "27017:27017"
    networks:
      - wanglab-net

  # 2. 主应用服务 (WangLab App)
  wanglab:
    build: 
      context: .
      dockerfile: Dockerfile
    container_name: wanglab-app
    restart: always
    ports:
      - "5000:5000" # 映射到宿主机 5000 端口
    environment:
      # 在 Docker 网络中，主机名直接写服务名 'mongodb' 即可
      - MONGO_URI=mongodb://mongodb:27017/wanglab
      - JWT_SECRET=wanglab_prod_secret_2025
      - API_KEY=${API_KEY} # 读取启动命令中的 API_KEY
    depends_on:
      - mongodb
    networks:
      - wanglab-net

networks:
  wanglab-net:
    driver: bridge

这个现象是生产环境部署（Production）与开发环境（Development）之间最典型的架构区别。
简单来说：在生产环境中，Node.js 后端不仅负责处理数据（API），还兼职充当了“Web 服务器”，把 React 生成的静态网页文件（HTML/CSS/JS）直接发送给浏览器。

为什么开发时是两个端口？
在开发阶段（npm run dev），你需要两个服务同时运行，是为了开发体验：
端口 A (Vite - 5173): 这是一个专门的开发服务器。它支持 热更新 (HMR)，你改一行代码，浏览器不用刷新就能立刻看到变化。它只负责前端。
端口 B (Express - 5000): 这是你的后端 API 服务器。
桥梁: Vite 通过配置 proxy，把你发往 /api 的请求悄悄转发给端口 B，让你感觉像是在同一个域下。

为什么部署后变成了一个端口？
在生产环境（部署到 NAS 或云服务器）时，我们不需要“热更新”功能了。我们将 React 代码打包（Build）成了静态文件（一堆 HTML、JS 和 CSS 文件）。
这时候，我们不需要 Vite 服务器了，只需要让 Express 后端多干一份活：
不仅处理 API 请求（比如 /api/login）。
还负责发送网页文件（当用户访问 / 时，把打包好的 index.html 给用户）。

代码是怎么实现的？
这就是我在 server/index.js 中添加的那几行关键代码的作用：
code
JavaScript
const path = require(‘path’);
const fs = require(‘fs’);

1. 告诉 Express：网页的静态文件（图片、JS、CSS）都在 ../dist 文件夹里
// 当浏览器请求 "assets/style.css" 时，Express 会直接从这里找并返回
const distPath = path.join(__dirname, '../dist');
app.use(express.static(distPath));


// ... API 路由定义 ...

// 2. “兜底”路由：处理 React 的页面跳转
// 如果请求的不是 /api 开头的（比如用户刷新了 http://host/notebook 页面），
// Express 不懂这是啥，但它会直接把 index.html 返回给浏览器。
// 浏览器收到 index.html 后，React 代码启动，自己接管路由，渲染出“实验笔记”页面。
app.get('*', (req, res) => {
  if (!req.path.startsWith('/api')) {
    const indexPath = path.join(distPath, 'index.html');
    if (fs.existsSync(indexPath)) {
        res.sendFile(indexPath);
    }
  }
});

这样做的好处是什么？
部署极其简单：你不需要配置 Nginx，也不需要分别启动两个容器。整个应用就是一个整体，丢到哪里都能跑。
彻底解决跨域问题 (CORS)：因为前端页面和后端 API 都在同一个端口（同一个域）下，浏览器不会拦截请求，不需要再做复杂的跨域配置。
节省资源：NAS 上少跑一个服务（Vite 开发服务器），节省内存和 CPU。
所以，变成一个端口说明你的应用已经是一个成熟的、可独立分发的全栈应用了！

1. 准备部署文件

# 在本地创建部署包
cd ~/Downloads/WangLab
mkdir -p nas-deployment
cd nas-deployment

# 创建 docker-compose.yml
cat > docker-compose.yml << 'EOF'
version: '3.8'

services:
  mongodb:
    image: mongo:4.4
    container_name: wanglab-mongodb
    restart: unless-stopped
    environment:
      MONGO_INITDB_ROOT_USERNAME: admin
      MONGO_INITDB_ROOT_PASSWORD: wanglab123
      MONGO_INITDB_DATABASE: wanglab
    volumes:
      - ./mongo-data:/data/db
    ports:
      - "15173:27017"
  
  wanglab:
    image: 你的用户名/wanglab:latest  # ← 修改这里
    container_name: wanglab-app
    restart: unless-stopped
    depends_on:
      - mongodb
    ports:
      - "5000:5000"
    environment:
      NODE_ENV: production
      MONGO_URI: mongodb://admin:wanglab123@mongodb:27017/wanglab?authSource=admin
      GOOGLE_API_KEY: ${GOOGLE_API_KEY}
    volumes:
      - ./logs:/app/logs
EOF

# 创建环境变量模板
cat > .env.example << 'EOF'
GOOGLE_API_KEY=你的Google API密钥
# 其他环境变量...
EOF

# 创建启动脚本
cat > start.sh << 'EOF'
#!/bin/bash
echo "🚀 启动 WangLab..."

# 创建目录
mkdir -p mongo-data logs

# 设置环境变量（如果存在 .env）
if [ -f .env ]; then
    export $(cat .env | xargs)
fi

# 启动
docker-compose up -d

# 等待
sleep 10

echo ""
echo "✅ 启动完成!"
echo "🌐 访问: http://$(hostname -I | awk '{print $1}'):5000"
echo "🔑 MongoDB: admin / wanglab123"
EOF

chmod +x start.sh

# 创建停止脚本
cat > stop.sh << 'EOF'
#!/bin/bash
echo "🛑 停止 WangLab..."
docker-compose down
echo "✅ 已停止"
EOF

chmod +x stop.sh

# 打包
cd ..
tar -czvf wanglab-deploy.tar.gz nas-deployment/

2. 上传到飞牛 NAS

# 上传部署包
scp wanglab-deploy.tar.gz admin@你的NAS-IP:/volume1/docker/

# 例如：
# scp wanglab-deploy.tar.gz admin@192.168.1.100:/volume1/docker/

3. 在飞牛 NAS 上部署

# SSH 连接到 NAS
ssh admin@你的NAS-IP

# 解压并部署
cd /volume1/docker
tar -xzvf wanglab-deploy.tar.gz
cd nas-deployment

# 修改镜像名为你的 Docker Hub 镜像
sed -i 's/你的用户名/实际用户名/' docker-compose.yml

# 配置环境变量
echo "GOOGLE_API_KEY=你的实际API密钥" > .env

# 启动
chmod +x *.sh
./start.sh

# 获取访问地址
hostname -I
# 输出类似：192.168.1.100

⚠️ 常见问题解决

问题1：端口被占用

# 查看占用进程
sudo lsof -i :5000
sudo lsof -i :15173

# 停止占用进程
sudo kill $(sudo lsof -ti:5000)

问题2：MongoDB 连接失败

# 在 docker-compose.yml 中确保：
environment:
  MONGO_URI: mongodb://admin:wanglab123@mongodb:27017/wanglab?authSource=admin

问题3：镜像构建失败

# 检查 Dockerfile 中的 Node.js 版本
# 应该使用：FROM node:20-alpine

# 清理缓存重新构建
sudo docker system prune -f
sudo docker build --no-cache -t my-wanglab:v1 .

问题4：缺少环境变量

# 创建 .env 文件
echo "GOOGLE_API_KEY=你的密钥" > .env

# 启动时指定
docker-compose --env-file .env up -d

💾 版本更新流程

当有代码更新时：

  1. 本地更新代码
  2. 重新构建镜像: sudo docker build -t my-wanglab:v2 .
  3. 推送到 Docker Hub: 
    sudo docker tag my-wanglab:v2 你的用户名/wanglab:v2
sudo docker push 你的用户名/wanglab:v2
  4. 在 NAS 上更新:
    cd /volume1/docker/nas-deployment
docker-compose pull
docker-compose up -d

📞 紧急恢复

如果部署失败：

  1. 查看详细错误:

    docker-compose logs --tail=100

  2. 回滚到上一个版本:

    # 修改 docker-compose.yml 中的镜像标签
# 从 :latest 改为 :v1
docker-compose up -d

  3. 完全重置:

    docker-compose down -v  # 删除数据和容器
rm -rf mongo-data logs  # 删除数据目录
./start.sh              # 重新启动

✅ 检查清单

部署前检查：

  • Dockerfile 使用 Node.js 20
  • 后端 package.json 中 @google/genai 版本正确
  • docker-compose.yml 中的镜像名已更新
  • 端口 5000 和 15173 未被占用
  • Google API Key 已准备好

部署后检查：

  • 两个容器都在运行 (docker ps)
  • 应用能正常访问 (http://NAS-IP:5000)
  • MongoDB 能连接 (telnet NAS-IP 15173)
  • 日志无报错 (docker-compose logs)
